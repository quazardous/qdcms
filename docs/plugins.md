# Plugin framework

> **Scope.** This document describes the plugin / migration / entity
> primitives that live in `@quazardous/qdcms-core` and power qdcms
> (qdcms-backend, qdcms-frontend, qdcms-admin). The framework is
> **agnostic** at the contract level — it knows nothing about blocks,
> layouts, admin pages, or any other CMS-specific concept — but it
> lives in the qdcms repo because qdcms is its primary consumer.
> Shared primitives (SignalBus, Stack, i18n) used by both qdcms and
> qdadm remain in `@quazardous/qdcore` (qdadm repo).

## 1. Overview

A **plugin** is a unit of distribution that can:
- contribute its own database tables (with a unique prefix)
- extend other plugins' tables (add columns)
- contribute application code (blocks, components, services — declared by
  consuming layers, not by qdcore)
- declare dependencies on other plugins with semver-range version
  constraints
- ship optional **upgrade hints** describing how data should evolve
  between versions of itself

The framework provides:
- A typed manifest format and validation
- A registry (with a minimal in-memory reference impl)
- A schema composer that merges multi-plugin contributions
- An ownership tracker for cross-plugin column extensions
- A migration runner that converges the live DB to the desired state
- Hints discovery and execution from per-plugin `upgrades/` directories

Built on top of **MikroORM 6.x** for the SQL layer (driver-agnostic;
SQLite/MariaDB/PostgreSQL).

## 2. Vocabulary

| term | meaning |
|---|---|
| **Plugin** | a unit of distribution (manifest + code + optional upgrade hints) |
| **PluginManifest** | the declarative description: id, version, prefix, dependencies, entities, extensions |
| **Plugin author** | the person/team publishing the plugin; ships manifest + code + hints |
| **Plugin consumer** | the site/app installing the plugin; owns the migration application and DB state |
| **Entity** | a business concept ("user", "post"); just a name |
| **EntityDescriptor** | runtime schema of an entity (fields, indexes) |
| **EntityRegistry** | typed collection of descriptors |
| **Repository\<T\>** | typed CRUD/query handle for one entity |
| **EntityManager** | façade routing to repos + cache + signals (host layer) |
| **BackendStorage** | schema-aware, transactional, server-side (Node) |
| **FrontendStorage** | API client + cache + invalidation, browser-side |
| **Driver** | low-level transport (better-sqlite3, mysql2, pg) |
| **Dialect** | SQL grammar variant (SQLite vs MariaDB vs Postgres) |
| **Upgrade hint** | author-provided guidance on how to transform data when moving between two versions of a plugin |
| **Hint file** | `upgrades/<target-version>.yaml` — declares the steps required to bring an instance up to that version |
| **MigrationRunner** | applies hints + structural diff, owned by the consumer instance |
| **MigrationStore** | per-instance persistence of "what's applied" (`qdcms_schema_state` table) |
| **Module** | qdadm-specific extension (frontend-only, in-process); LIGHTER than a plugin |

**Terms to NOT use** (ambiguous): *Source*, *Provider*, *Connector*,
*Adapter* (only as adjective, never as a layer name).

## 3. Actors — author vs consumer

**Plugin author** writes the manifest, ships the code, optionally
provides hint files. Does NOT know who will install the plugin. Does
NOT maintain pre-baked migration SQL — migrations are computed by each
consumer instance.

**Plugin consumer** chooses to install (or not) a plugin on their
site. Has their own `qdcms_schema_state` per site. Owns the migration
SQL: it is generated by the consumer's runner from manifests + hints,
applied locally, recorded in the consumer's DB.

```
   ┌──────────────────────┐    publish    ┌─────────────────────┐
   │  Plugin author       │ ────────────► │  Distribution       │
   │  (manifest + code +  │               │  (npm / git / ...)  │
   │   optional hints)    │               │                     │
   └──────────────────────┘               └──────────┬──────────┘
                                                     │ install
                                                     ▼
                                         ┌─────────────────────┐
                                         │  Plugin consumer    │
                                         │  (qdcms instance,   │
                                         │   computes + applies│
                                         │   migrations,       │
                                         │   tracks state)     │
                                         └─────────────────────┘
```

**Implication: migrations are per-instance.** Two consumers running
the same plugin at the same version may have applied different
migration SQL (because their starting DB state differed, or because
they customised templating-driven config). The plugin author has no
visibility — and no responsibility — for the actual SQL run on each
site.

## 4. Layering

### 4.1 `@quazardous/qdcms-core` — agnostic primitives

| subpath | content |
|---|---|
| `/entity` | EntityDescriptor, EntityRegistry, Repository, BackendStorage, FrontendStorage (contracts) |
| `/plugin` | PluginManifest, Plugin, PluginRegistry contract, **InMemoryPluginRegistry** reference impl, validateManifest |
| `/migration` | Migration, MigrationContext, MigrationRunner contract, hashSchema, composeSchema, OwnershipTracker (Node-only) |
| `/sql` | MikroOrmBackendStorage, SqlMigrationStore, MikroOrmMigrationRunner (concrete SQL impl, Node-only) |

This layer:
- has zero knowledge of CMS / admin domains (the contracts could
  serve any plugin-based app)
- depends on `@quazardous/qdcore` for SignalBus, Stack, i18n
  (truly cross-app primitives)
- ships a minimal in-memory registry and a working SQL runner
- is what tests and small apps consume directly

### 4.2 `qdcms-backend` (future) — rich impl

When qdcms-backend lands as a separate package, it will provide:
- `YamlDbPluginRegistry` — reads "available plugins" from
  `qdcms.config.yaml`, reads enable/disable overrides from a DB table
  (admin-UI manageable), composes both into the in-memory shape
- `PluginOrchestrator` — high-level operations: "enable plugin X"
  (cascades to deps), "disable plugin X" (refuses if depended on),
  with admin-UI preview support
- Cascade install / uninstall logic (see section 11)

**The base registry has no concept of activation.** A plugin that's
referenced (registered) is, for the base impl, simply present and
operational. The active/inactive distinction is a richer concept that
qdcms-backend's registry will introduce; qdcore stays out of it.

## 5. Plugin manifest

```yaml
# plugins/dynamic-content/qdcms-plugin.yaml
id: dynamic-content
version: 0.1.0
prefix: dc
title: Dynamic Content
description: Custom post types and field UI
dependencies:
  - id: core
    version: '^1.0.0'   # semver range, npm-style
entities:
  posts:
    fields:
      id: { type: uuid, pk: true }
      title: { type: string, length: 255 }
      created_at: { type: timestamp, default: now }
extensions:
  core_users:
    bio: { type: text, nullable: true }
schemaManaged: true   # default, can be set to false
```

The manifest describes the **current** state of the plugin only. No
historical baggage, no inline upgrade annotations — those live in the
`upgrades/` directory (see section 9).

### 5.1 Naming rules

| field | regex | notes |
|---|---|---|
| `id` | `^[a-z][a-z0-9_-]*$` | lowercase, dashes OK |
| `prefix` | `^[a-z][a-z0-9_]*$` | lowercase, **no dashes** (would break table naming on some dialects) |
| `version` | strict semver `MAJOR.MINOR.PATCH[-prerelease][+build]` | validated at register time |

### 5.2 Field types

| type | mapped to (SQLite) | mapped to (MariaDB) | mapped to (Postgres) |
|---|---|---|---|
| `uuid` | TEXT | CHAR(36) | UUID |
| `string` | TEXT | VARCHAR | VARCHAR |
| `text` | TEXT | TEXT | TEXT |
| `integer` | INTEGER | INT | INTEGER |
| `bigint` | INTEGER | BIGINT | BIGINT |
| `float` | REAL | DOUBLE | DOUBLE PRECISION |
| `boolean` | INTEGER | TINYINT | BOOLEAN |
| `json` | TEXT | JSON | JSONB |
| `date` | TEXT | DATE | DATE |
| `datetime` | TEXT (ISO-8601) | DATETIME | TIMESTAMP |
| `timestamp` | TEXT (ISO-8601) | DATETIME | TIMESTAMP |

Escape hatch: `dbType: 'tsvector'` overrides the portable type when
dialect-specific features are needed (breaks portability — caller must
declare `requiresDialect`).

### 5.3 schemaManaged

- `true` (default) — qdcms manages the plugin's schema. Migrations
  computed and applied at install/upgrade, rolled back at uninstall.
- `false` — qdcms does **not** touch the schema. Use for plugins that
  integrate with an externally-managed backend. Lifecycle state is
  still tracked.

## 6. Lifecycle states

```
'registered'  ← manifest loaded, NO db work yet
'installed'   ← runner.install OK, schema applied
'failed'      ← a transition threw; intervention required

(reserved for richer impls — qdcore never sets these)
'active'      ← (future) admin-enabled; code loaded, hooks running
'inactive'    ← (future) installed but admin-disabled; schema kept
```

**Important.** The base `InMemoryPluginRegistry` only ever sees
`'registered'`, `'installed'`, and `'failed'`. The richer `'active'` /
`'inactive'` values exist in the type union for forward compatibility
with qdcms-backend's `YamlDbPluginRegistry`. Do not introduce code in
qdcore that branches on those values.

## 7. Dependencies and version ranges

```yaml
dependencies:
  - id: core
    version: '^1.0.0'   # any 1.x.y, not 2.0.0
  - id: i18n
    version: '~2.3.0'   # any 2.3.x, not 2.4.0
  - id: storage
    version: '>=3.0.0 <4.0.0'   # explicit range
  - id: monitoring
    # version omitted == '*' (any version)
```

**Single version per site.** Unlike npm (which can install multiple
copies of the same package), qdcms enforces ONE installed version per
plugin. If two plugins require incompatible ranges of a third, the
installation fails with a clear error — the consumer must reconcile
manually (upgrade/downgrade, or forgo one of the plugins).

**No auto-resolution.** qdcore does NOT attempt to fetch missing
dependencies. The consumer's plugin set is what it is; missing deps
throw at `resolveOrder()` time.

**Validation flow:**
1. At register time: `validateManifest()` ensures version ranges are
   valid semver ranges (rejects `'wat'`, accepts `'^1.0.0'`).
2. At resolve time: `resolveOrder()` runs `semver.satisfies()` on each
   dep against the installed version; throws on mismatch.

## 8. Schema management

### 8.1 Table naming

A plugin owns tables prefixed with its `prefix_`. The composer applies
the prefix automatically (idempotent — entities whose `tableName`
already starts with the prefix are not double-prefixed).

```yaml
# plugin id=dynamic-content, prefix=dc
entities:
  posts:                # ⇒ physical table dc_posts
    tableName: posts    # could also be `dc_posts` directly
```

System tables created by qdcore use the `qdcms_` prefix
(`qdcms_schema_state`).

### 8.2 Extensions

A plugin can add columns to another plugin's table by listing the
foreign physical table name under `extensions:`:

```yaml
extensions:
  core_users:
    newsletter_opt_in:
      type: boolean
      nullable: true
    bio:
      type: text
      nullable: true
```

**Rules:**
- The extending plugin MUST declare the owning plugin in its
  `dependencies` (validated by `registry.validateExtensions()`).
- A plugin cannot extend its own table — declare the fields on the
  entity directly.
- Two plugins cannot add the same column name to the same table —
  composer throws `MigrationOwnershipError`.

### 8.3 Ownership tracking

Each field in the composed schema carries an `owner` stamp (either the
plugin owning the table, or — for extensions — the plugin adding the
column). On uninstall:
- For OWNED tables: the entire table is dropped.
- For EXTENSION columns: only those columns are dropped; the foreign
  table and other plugins' columns remain.

The `OwnershipTracker` in `qdcore/migration` keeps an in-memory mirror
of `<plugin → tables, columns>` for audit and admin-UI use. The
runtime DDL is generated from the composed schema, not from the
tracker — they are kept in sync by construction.

## 9. Migration model

**Migrations are per-instance, computed by the consumer.**

The plugin author does NOT ship pre-baked SQL files. Each consumer
site computes the SQL needed to bring its own DB to the state declared
by the plugin's manifest, optionally guided by hints from the author.

### 9.1 What the author ships

In the plugin distribution:

```
plugins/dynamic-content/
  qdcms-plugin.yaml          ← current manifest (state of vN.M.K)
  upgrades/                  ← OPTIONAL hints directory
    1.0.0.yaml               ← hints for transitioning TO 1.0.0
    1.5.0.yaml               ← hints for transitioning TO 1.5.0
    2.0.0.yaml               ← hints for transitioning TO 2.0.0
    helpers/
      split-tags.ts          ← TS scripts referenced from YAML hints
  src/                       ← plugin code (blocks, components, ...)
```

The `upgrades/` directory is **optional**. Plugins that have never
needed data-aware migrations don't have one — the consumer just runs
structural diff.

### 9.2 Hint file naming

Convention: `<target-version>.yaml` in `upgrades/`. The filename IS
the target version; the source version is implicit (whatever was
installed before).

```
upgrades/
  0.5.0.yaml      ← applies when transitioning TO 0.5.0
  1.0.0.yaml      ← applies when transitioning TO 1.0.0
  2.0.0-beta.1.yaml
  2.0.0.yaml
```

Pre-release versions are supported (semver-aware ordering).

### 9.3 Hint file format

```yaml
# upgrades/2.0.0.yaml
description: |
  Renames title→headline, adds author_id with backfill,
  drops legacy_count.

# Optional safety guard. If present, the upgrade is REFUSED when the
# instance is below this version. Forces the consumer to apply
# intermediate upgrades first.
min_version: '1.5.0'

steps:
  - rename_field:
      entity: posts
      from: title
      to: headline

  - add_field:
      entity: posts
      field: author_id
      backfill_from: legacy_author_id
      backfill_default: '00000000-0000-0000-0000-000000000000'

  - drop_field:
      entity: posts
      field: legacy_count

  - script: ./helpers/split-tags.ts
    description: 'Extract embedded tags into post_tags table'
```

Three top-level fields:
- `description` — human-readable, optional, recommended
- `min_version` — semver version (not a range), optional safety guard
- `steps` — required, the actual work

### 9.4 Step types

| step | sql equivalent (sqlite example) |
|---|---|
| `rename_field: { entity, from, to }` | `ALTER TABLE x RENAME COLUMN a TO b` |
| `add_field: { entity, field, backfill_from?, backfill_default?, backfill_sql? }` | `ALTER TABLE x ADD COLUMN ; UPDATE ...` |
| `drop_field: { entity, field }` | `ALTER TABLE x DROP COLUMN a` |
| `change_type: { entity, field, cast? }` | `ALTER TABLE x ALTER COLUMN ...` (dialect-specific) |
| `rename_table: { from, to }` | `ALTER TABLE a RENAME TO b` |
| `add_index: { entity, fields, unique? }` | `CREATE [UNIQUE] INDEX ...` |
| `drop_index: { entity, name }` | `DROP INDEX ...` |
| `script: ./path/to/file.ts` | run TypeScript module exporting `upgrade(ctx)` |

The `add_field` step picks up the field DEFINITION from the current
manifest (it doesn't redeclare type/length/nullable here). What the
hint provides is just the data-aware part: how to populate the new
column on existing rows.

The `script:` escape hatch loads a TS module exposing
`async upgrade(ctx: UpgradeContext)`. Context provides:
- `ctx.exec(sql, params?)` — run raw SQL
- `ctx.repo<T>(name)` — typed repository
- `ctx.dialect` — current SQL dialect (for conditional behaviour)
- `ctx.logger` — structured log into the migration trace

### 9.5 Resolution algorithm

```
upgrade plugin from current_version to target_version:
  candidates = files in upgrades/
             . parse semver from filename
             . filter v where current_version < v <= target_version
             . sort semver ascending

  in a transaction, for each version v in candidates:
    file = upgrades/<v>.yaml
    if file.min_version is present:
      if not satisfies(current_running_state, file.min_version):
        ABORT with explicit error
    apply file.steps in order
    record(plugin, hash(rendered_schema), plugin_version=v) in qdcms_schema_state
    set current_running_state = v

  # safety net: structural diff after all hints
  diff = SchemaComparator.compare(current_db, manifest_v(target_version))
  if diff has destructive operations not justified by drop_field steps:
    warn admin (or fail in strict mode)
  apply diff (typically empty if hints covered everything)
```

### 9.6 Edge cases

**First install (no prior version)** — no hints applied. Pure
structural diff creates all tables. The first row in
`qdcms_schema_state` records the initial install with
`plugin_version = manifest.version`.

**Skip-version upgrade** — instance at 0.5.0, target 2.0.0. Runner
applies all hint files in semver range (0.5, 2.0]. Versions without
a hint file are skipped (no transition needed beyond structural diff).

**No upgrades/ directory** — every transition uses structural diff
only. The consumer admin-UI surfaces destructive operations for
explicit confirmation.

**Plugin without `min_version` set anywhere** — runner attempts every
upgrade in sequence. If something fails (e.g., a step references a
field that doesn't exist), the transaction rolls back, the runner
reports the error to the admin, and the instance stays on its current
version.

**`min_version` blocks an upgrade** — runner reports:

```
Cannot upgrade plugin "dynamic-content" to 2.0.0:
  hint file 2.0.0.yaml requires plugin to be at version >= 1.5.0,
  but current installed version is 1.0.0.
  No intermediate upgrade file is available to bridge the gap.
  Contact the plugin author or upgrade through compatible
  intermediate releases.
```

### 9.7 qdcms_schema_state schema

```
qdcms_schema_state
  plugin            TEXT NOT NULL
  hash              TEXT NOT NULL          ← sha256 of rendered schema after this transition
  applied_at        TEXT NOT NULL          ← ISO-8601
  plugin_version    TEXT NOT NULL          ← plugin version after this transition
  upgrade_file      TEXT                   ← name of hint file applied (NULL if structural diff only)
  rendered_schema   JSON                   ← snapshot for the next diff baseline
  applied_sql       TEXT                   ← SQL executed (audit trail)
  PRIMARY KEY (plugin, hash)
```

One row per state-changing operation. The latest row's
`plugin_version` tells the runner where the instance currently is.
`rendered_schema` is the baseline for the next compose/diff cycle.

### 9.8 Validation at the author side

A future `qdcms-cli plugin:validate` command will check that:
- All hint files parse cleanly
- All `step` types are recognised
- `rename_field.from` references a field present in the prior state
- `rename_field.to` matches a field in the manifest
- `min_version` requirements are reachable through the hint chain
- Referenced TS scripts exist

Run by the plugin author in CI before releasing. Not a runtime
requirement on the consumer side, but reduces broken-on-arrival
plugins.

## 10. Storage layer split

| | BackendStorage | FrontendStorage |
|---|---|---|
| **runs in** | Node server (`qdcms-backend`) | Browser (`qdcms-frontend`, qdadm) |
| **knows the schema?** | yes (entity descriptors → SQL) | no (opaque API responses) |
| **transactions?** | yes | n/a |
| **persistence** | the DB itself | per-session cache (memory by default) |
| **invalidation** | n/a | signal-driven (SignalBus subscriptions) |
| **typical impl** | MikroOrmBackendStorage | (future) ApiFrontendStorage |

The two contracts are **separate** — they have different shapes
because they solve different problems. Don't try to unify them under
"Storage" — explicit `BackendStorage` / `FrontendStorage` everywhere.

## 11. Cascade enable/disable + cascade upgrade (future, qdcms-backend)

When qdcms-backend lands, it will introduce a `PluginOrchestrator`
above the runner with high-level operations:

### `enable(B)` — cascade install + activate
- Recursively enables B's dependencies (depth-first)
- For each dep already in `'active'` state: skip
- For each dep in `'inactive'` state: reactivate
- For each dep in `'registered'` state: install + activate
- Then install + activate B
- Version constraint check runs at every step; cascade does not
  bypass incompatibilities

### `disable(A)` — refuse if depended on (asymmetric)
- If any other plugin in `'active'` state depends on A: refuse
- The admin must disable dependents first, in reverse topological
  order

### `upgradePlan(plugin, targetVersion)` — cascade upgrade preview
- Computes which other installed plugins would have their dependency
  ranges violated by the target version
- For each affected plugin, finds compatible versions (if available)
- Returns a complete plan: `{ plugin → newVersion }` chain
- Throws if any plugin has no compatible version available

### `previewEnable(id)` / `previewUpgrade(id, version)` — admin-UI previews
- Both return a structured diff of what would happen
- Admin UI displays before executing
- Same primitives drive both single-plugin and cascade scenarios

**These belong to qdcms-backend, NOT qdcore.** qdcore exposes the
primitive `runner.install(one)` / `runner.uninstall(one)` /
`runner.upgrade(one)` operations; qdcms-backend composes them.

## 12. Multi-plugin scenarios

### 12.1 Conflict matrix

| collision | detected at | error |
|---|---|---|
| two plugins same id | register | PluginConflictError |
| two plugins same prefix | register | PluginConflictError |
| extension targets unknown table | resolveOrder / install | PluginDependencyError / MigrationOwnershipError |
| extension to other plugin's table without declared dep | validateExtensions | PluginDependencyError |
| two plugins extend same table with same column | composeFullSchema (install) | MigrationOwnershipError |
| dependency cycle A → B → A | resolveOrder | PluginDependencyError |
| dependency missing | resolveOrder | PluginDependencyError |
| dependency version range not satisfied | resolveOrder (Phase: pending) | PluginDependencyError |
| upgrade `min_version` not satisfied | runner.upgrade (Phase: pending) | MigrationError |
| cascade upgrade has no compatible version | qdcms-backend orchestrator (Phase 3) | OrchestrationError |

### 12.2 Dependency ordering

`InMemoryPluginRegistry.resolveOrder()` returns plugin ids in
topological order (every plugin appears after all its dependencies).
DFS with three-color marking (white/gray/black) detects cycles. The
runner installs in this order; uninstalls in reverse order.

## 13. Testing posture

The qdcore test suite is split:

- **Pure-function tests** (`tests/plugin/`, `tests/migration/`) — no
  DB, run instantly. Cover hash determinism, manifest validation,
  registry conflict detection, dependency sort, OwnershipTracker
  conflicts, schema composition.
- **Integration tests** (`tests/sql/`) — temp-file SQLite per test,
  full MikroORM stack. Cover install/uninstall lifecycle,
  multi-plugin extensions with selective drop, data preservation
  across schema changes, dependency ordering enforcement, idempotence,
  reinstall, runtime conflict detection.

**Why temp file, not `:memory:`?** SQLite memory databases are
per-connection. The runner disconnects/reconnects on every schema
change (MikroORM v6 metadata is fixed at init), which would wipe a
`:memory:` DB between operations and break any data-preservation test.
